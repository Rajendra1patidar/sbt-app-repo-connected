# Deploying SBT Sales (Netlify + Render + MongoDB Atlas) — all free tier

Three pieces, deployed separately:

- **Database** → MongoDB Atlas (free M0 cluster)
- **Backend** (`/backend`) → Render (free web service)
- **Frontend** (`/frontend`) → Netlify (free static hosting)

## 1. Database — MongoDB Atlas

1. Create a free account at mongodb.com/cloud/atlas, create a free **M0** cluster.
2. Database Access → add a user with a username/password (save these).
3. Network Access → Add IP Address → **Allow access from anywhere** (`0.0.0.0/0`) — needed since Render's IPs aren't static on the free plan.
4. Connect → Drivers → copy the connection string. It looks like:
   `mongodb+srv://<username>:<password>@<cluster>.mongodb.net/sbt?retryWrites=true&w=majority`
   Replace `<username>`/`<password>`, keep `/sbt` as the database name.

## 2. Backend — Render

The repo already has `render.yaml` at the root and `backend/railway.json` (harmless if you don't use Railway).

1. Push this repo to GitHub (with the changes from this session).
2. On Render: **New → Blueprint**, point it at the repo. Render reads `render.yaml` and sets up the service with root dir `backend`.
3. When prompted, fill in the env vars it flags as `sync: false`:
   - `MONGODB_URI` — the Atlas connection string from step 1
   - `CORS_ORIGIN` — leave blank for now, come back after step 3 (Netlify) with your Netlify URL
4. `JWT_SECRET` is auto-generated by the blueprint — no action needed.
5. Deploy. Note your backend URL, e.g. `https://sbt-backend.onrender.com`.
6. Sanity check: open `https://sbt-backend.onrender.com/api/health` — should return `{"status":"ok",...}`.

*(No Blueprint access, or prefer manual setup? New → Web Service → root dir `backend` → build `npm install` → start `node server.js`, then add the same three env vars by hand.)*

Note: Render's free tier spins the service down after ~15 min idle; the first request after that takes ~30–50s to wake up. Fine for a small internal tool, worth knowing so it doesn't look "broken" on first load.

## 3. Frontend — Netlify

1. On Netlify: **Add new site → Import from Git**, pick this repo.
2. **Base directory**: `frontend` (important — this is a monorepo). Netlify will then use `frontend/netlify.toml` for build command (`npm run build`) and publish dir (`dist`).
3. Site settings → Environment variables → add:
   - `VITE_API_URL` = your Render URL from step 2, e.g. `https://sbt-backend.onrender.com`
4. Deploy. Note your Netlify URL, e.g. `https://your-app.netlify.app`.

## 4. Close the loop — CORS

Go back to Render → your backend service → Environment → set:
```
CORS_ORIGIN=https://your-app.netlify.app
```
(comma-separate multiple origins if needed, e.g. add `http://localhost:5173` for local dev). Redeploy the backend.

## 5. First run

Open the Netlify URL. The app now shows a real sign-in screen (backed by the API, not the old browser-only PIN lock):
- First time: tap **"Create an account"**, set an email + PIN (4+ digits).
- After that: sign in with the same email + PIN. Session persists in `localStorage` as a JWT.
- Sign out via the button at the bottom of the sidebar.

All data (customers, items, orders, quotes, invoices, challans, expenses, payments, settings) now lives in MongoDB and survives refreshes/devices — it's no longer lost on reload.

## Local development

```bash
# backend
cd backend
cp .env.example .env   # fill in MONGODB_URI, JWT_SECRET, CORS_ORIGIN=http://localhost:5173
npm install
npm run dev             # http://localhost:5000

# frontend
cd frontend
cp .env.example .env    # VITE_API_URL=http://localhost:5000
npm install
npm run dev              # http://localhost:5173
```
